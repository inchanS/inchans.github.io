플랫폼 이전

move to github + hugo

before: google blogger

현재 바로 이 블로그의 플랫폼이 google blogger이다.
tistory나 naver 블로그에 비해 내가 원하는 문법의 허용범위가 더 커서 오랫동안 사용중이었다.
(예를 들면 toc, 각주, MarkDown 문법과 HTML 문법으로의 글 작성 모두 지원 등…)

기술 블로그에서 가장 유용한 부분은 toc(table of contents)라고 생각한다.
header태그를 이용한 toc의 가장 큰 장점은 문서 하나의 크기가 클때, 해당 헤더 부분만의 링크를 가질 수 있다는 점, 이로써 참고 링크에서 다시한번 해당 부분을 찾는 번거로움을 줄일 수 있다.1

문제는 바로 이 toc를, google blogger에서 문서 상단에 고정된 방식이 아니라,
스크롤 중에도 상시 표현되는 사이드 바로 빼고 싶었는데 그러자니 xml을 건드리는 수고가 여간 큰게 아니었다.

게다가 google blogger의 xml 방식 때문인지는 모르겠지만 website 최적화 체감지수가 미묘하게 불쾌하달까? 반응이 반템포 느리다.

체험해본바 주관적인 최적화 속도는 다음 순서였다.

github.io >= velog > tistory » naver blog > google blogger

after: github.io http://github.io

보통 github 블로그 구성시 많이들 이용하는 정적사이트 생성기로서 ruby 언어 기반의 jekyll이 있는데, hugo를 택한 이유는 다음과 같다.

Jekyll
현재 가장 인기 있음(깃헙 별 수 제일 많음)
한글 레퍼런스도 제일 많음
느리다는 제보가 많음(몇 십개의 포스팅 뿐인데도 빌드 하는데 5분씩 걸린다고)
윈도우 공식 지원 안됨

Hexo

자바스크립트(Node.js) 기반
한글 레퍼런스 꽤 많음
메인 개발자가 손을 놓은 듯
개발자가 중국인? 이라 구글링하면 중국어 글이 많이 나옴

Hugo
Go 기반
빌드 빠름
문서화 잘돼있음
깃헙 별 수가 헥소보다 많음
한글 레퍼런스가 거의 없음

출처: http://tadakichi.tistory.com/188

pros

  • 사용자 설정이 자유롭다.
  • 개발자들의 기술블로그로써 어울리는 다양한 테마가 있다.
  • git으로 관리가 쉽다?(개발자의 경우)
  • hugo server 명령어를 통해 web과 동일한 수준의 전체 site map 단의 미리보기 가능 (댓글처럼 외부 콤포넌트를 이용한 부분은 제외)
  • 게시물의 title, description등을 따로이 설정 가능함으로 SEO에 유리하다.
  • 컨텐츠의 원본 문서를 언제든 HTML로 변환할 수 있는 MD파일이 곧 게시글 초안이자 발행 문서임으로, 해당 플랫폼 귀속이 아닌 내 로컬 스토리지에서 관리가 가능하다.
  • 지금처럼 언제든 플랫폼을 자유롭게 이동하며 전체 컨텐츠를 재등록하기가 수월하다.
  • 이미지의 경우, 로컬+github 이중 관리로 image 삽입시, 링크소실의 염려나 원본 이미지 파일의 관리 등이 수월하다.
    • (클라우드 업로드 후, image src 방식 사용시 클라우드 서비스가 종료하면서 상당히 머리 아픈 일이 생긴 경험이 있다;;)
  • notion, obsidian의 기능중 하나인 callout 기능을 손쉽게 구현 가능
    • hugo에서는 이를 admonition이라고 부른다.2
    • hugo의 admonition의 특징은 toggle이 default 옵션이라는 점!!

cons

  • 장점에서 언급됐었지만, 아이러니하게도 git으로 관리해야한다는 점
  • hugo의 경우, 비공개 게시글이나 비공개 초안 등을 관리한다면 2개의 repository를 운영해야한다.
  • 이는 하나의 게시글당 2번의 commit과 push가 필요하다는 뜻
Tip
하지만, 쉘스크립트를 이용해 한번에 해결할 수 있다!!

comment

google blogger에서는 google 댓글을 사용했었는데 github issue를 이용한 utterances를 설치하여 댓글로 자리잡음 하였다.
light, dark theme를 auto로 설정하여 적용시킬 수 있어 외부 컴포넌트임에도 꽤 이질감 없는 디자인을 보여준다.
개발자들이라면 꼭 가지고 있는 github 계정을 활용한 댓글 방식이라, 기술 블로그다보니 조금이나마 더 나은 전문성과 투명성을 업은 양질의 댓글 문화를 기대할 수 있다.

원래 현재 사용중인 이 도메인(codject.dev)은 프로젝트에서 쓰려고 만든 도메인이었는데, 어떻게 하다보니 여기에서 쓰고 있었다.
inchan.dev 도메인은 작년 justcode 부트캠프를 시작하면서 구매해놓았었는데 이제서야 내 이름을 걸고 만든 도메인으로 기술 블로그를 운영해볼까 하는 생각이 들었다.

search service : algolia

SaaS모델로서 웹 검색을 제공하는 서비스
참고문서 https://www.algolia.com/doc/: https://www.algolia.com/doc/https://www.algolia.com/doc/
Algolia는 검색 구현과 검색 분석의 두 부분으로 구성되어 있습니다. 구현 도구를 사용하면 개발자가 사용자를 위한 훌륭한 검색 환경을 더 쉽게 만들고 유지할 수 있습니다. https://www.algolia.com/doc/guides/getting-started/what-is-algolia/
pros & cons
pros

####### json 파일을 활용한 index 방식이라 그런지 속도가 굉장히 빠르다.

####### 검색하고자 하는 문서의 범위를 사용자가 세부적으로 조정 가능하다.

####### 검색어 관련 세부적인 분석이 가능하다.

####### 소규모 사이트의 경우 free plan으로도 어느정도 커버 가능하다.

cons

####### npm - push_algolia_json.js 파일을 실행시키는 스크립트를 만들고 새로운 문서의 발행시마다 API를 통해 indexing을 실행해줘야 하다. (algolia에서 자동으로 robots.txt 파싱하는 방식이 아닌듯…)

####### 해결 - 스크립트를 이용해 새 문서 발행시 algolia index 과정을 자동화시킴으로써 누락과 과정의 번거로움을 간소화!

obsidian 연동
obsidian내 편입시킴으로써, 모든 개인 기술 문서를 통합 관리 가능
이미지 첨부시 obsidian의 자동 링크연결 기능 활용(drag & drop으로 굉장히 편하다.)
